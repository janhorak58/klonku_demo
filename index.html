BP Checker — API Design

Návrh FastAPI backendu pro službu automatické kontroly bakalářských a diplomových prací pomocí LLM.
Platforma: TUL (FM, FE, FP)  |  Verze: 0.1 draft  |  Datum: 2026-03-31

Architektura systému

Systém se skládá ze dvou oddělených aplikací: Flask frontendu, který obsluhuje UI, a FastAPI backendu, který zajišťuje veškerou business logiku, správu dat a orchestraci LLM pipeline. Frontend volá backend přímo přes REST API pomocí JWT tokenů.

Flask Frontend

Renderování UI, session management pro prohlížeč, předává JWT tokeny z backendu.

FastAPI Backend

Veškerá business logika, autentizace, správa projektů, spouštění pipeline.

ARQ + Redis (Task queue)

Asynchronní zpracování dokumentů. ARQ worker konzumuje joby z Redis fronty. Polling z frontendu zjišťuje stav analýzy. Priority: paid > free.

Pandoc (Konverze)

Převod DOCX a LaTeX na Markdown před analýzou. Při selhání konverze dostane verze stav conversion_failed.

Lokální filesystem (soubory)

Nahrané soubory a Markdown výstupy jsou uloženy na lokálním disku. Constraint: vyžaduje single-instance deploy — FastAPI backend a ARQ worker musí běžet na stejném hostu (sdílený volume v docker-compose). Pokud by se v budoucnu přešlo na více instancí, je nutné migrovat na S3-compatible storage (MinIO nebo AWS S3).

LLM

Každý krok pipeline volá LLM s příslušným promptem. Výstupem je standardizovaný JSON chyb. Dokument se do LLM posílá jako user content oddělený od system instrukce.

Auth flow

Backend vydává JWT access token (krátká životnost, ~15 min) a refresh token (delší životnost). Flask frontend ukládá tokeny a přikládá Authorization: Bearer <token> ke každému API volání.

sequenceDiagram participant U as Uživatel participant F as Flask Frontend participant A as FastAPI Backend participant G as Google OAuth U->>F: Klikne "Přihlásit se" alt Email + heslo F->>A: POST /auth/login A-->>F: access_token + refresh_token else Google OAuth F->>G: Redirect na Google G-->>F: OAuth callback s code F->>A: POST /auth/google {code} A-->>F: access_token + refresh_token end F-->>U: Uloží tokeny, přesměruje na dashboard Note over F,A: Každý následující request F->>A: GET /projects (Authorization: Bearer token) A-->>F: data Note over F,A: Token expiruje F->>A: POST /auth/refresh {refresh_token} A-->>F: nový access_token
Refresh token musí být uložen bezpečně (httpOnly cookie nebo bezpečné úložiště na serveru Flask frontendu — nikdy v localStorage).
Rate limiting na auth endpointech: POST /auth/login a POST /auth/register jsou limitovány na 10 pokusů / minutu per IP. Po překročení limit vrátí 429 Too Many Requests s hlavičkou Retry-After. Implementace: FastAPI middleware (např. slowapi s Redis backendem).
CORS: Backend povoluje requesty pouze z origin adresy Flask frontendu. Nastavení přes CORSMiddleware — allow_origins je konfigurovatelné přes env proměnnou, nikoli hardcoded.
HTTPS: Veškerá komunikace (Browser → Flask i Flask → FastAPI) musí probíhat přes HTTPS. Interní Flask → FastAPI komunikace může být HTTP v rámci privátní sítě/Docker networku, ale nikdy přes veřejný internet.

Datový model (ER diagram)

Multi-tenant architektura: každý student (User) má vlastní projekty (Project). Každý projekt patří jedné fakultě a má sérii verzí (DocumentVersion) — snapshot při každém uploadu. Každá verze má výsledky analýzy (FoundError). Pipeline se konfiguruje per-fakulta superadminem.

Poznámky k modelu (v. 0.2):
  • USER: odstraněny sloupce tokens_used_today/week — nahrazeny tabulkou TOKEN_USAGE_EVENT (atomické operace, auditovatelnost). Přidán tier (free/paid).
  • PIPELINE_STEP: přidán prompt_version + entita PIPELINE_STEP_VERSION — zachovává historii promptů pro reprodukovatelnost výsledků.
  • FOUND_ERROR: odstraněn redundantní error_type_id (FK je error_type_slug). Přidán context_snippet (~50 znaků okolí chyby) pro robustní diff mezi verzemi. Sémantika is_resolved: příznak je nastaven na záznamu ze staré verze a říká „tato chyba byla opravena v novější verzi". Nová verze neopravených chyb obsahuje odkaz přes previous_error_id.
  • DOCUMENT_VERSION: markdown_content přesunuto do souborového systému, v DB jen cesta markdown_path.
  • ADMIN_AUDIT_LOG: logování všech admin akcí (změny promptů, úpravy uživatelů, atd.).

Upřesnění datového modelu:
  • FOUND_ERROR.error_type_slug jako FK je riziková volba. Textový cizí klíč znamená, že přejmenování slugu v ERROR_TYPE rozbije historické záznamy. Správné řešení: FK je error_type_id UUID, slug je jen denormalizovaná kopie pro rychlé čtení. Při změně slugu se provede UPDATE FOUND_ERROR SET error_type_slug = 'new' WHERE error_type_id = '...'.
  • USER tabulka — nová pole: přidat monthly_scan_count INT DEFAULT 0 a monthly_scan_budget INT. Odstranit daily_token_budget a weekly_token_budget (nahrazeno scan metrikou).
  • Chybějící entita v ER diagramu: ALLOWED_DOMAIN (viz sekce Institucionální emaily) — přidat do diagramu v příští revizi.

Autentizace

Veřejné endpointy pro přihlášení, registraci a správu tokenů.

POST /auth/register Registrace email + heslo public
POST /auth/login Přihlášení email + heslo, vrátí JWT public
POST /auth/google OAuth callback — vymění Google code za JWT public
POST /auth/refresh Vydá nový access token pomocí refresh tokenu public
POST /auth/logout Invaliduje refresh token (blacklist) auth
GET /auth/verify-email/{token} Potvrzení emailu (UUID token z verifikačního mailu, platnost 24 h) public
POST /auth/forgot-password Odeslání odkazu pro reset hesla na zadaný email public
POST /auth/reset-password Reset hesla pomocí tokenu z emailu (token + nové heslo) public
Refresh token blacklist: Refresh tokeny jsou po odhlášení ukládány do Redis setu s TTL odpovídajícím jejich zbývající platnosti. Při každém POST /auth/refresh backend zkontroluje, zda token v setu není. Při výpadku Redisu selže endpoint s 503 — nikdy nesmí propadnout na "token je platný". DB fallback (tabulka REVOKED_TOKEN) se použije pouze pokud Redis není dostupný jako primární store.
Google OAuth — CSRF ochrana: Před přesměrováním na Google vygeneruje Flask state token (náhodný UUID), uloží ho do session a předá jako parametr OAuth URL. Google ho vrátí v callbacku — Flask ověří shodu před předáním code na FastAPI. Bez toho je flow zranitelné vůči CSRF útoku.

Verze dokumentů

Každý upload souboru vytvoří nový snapshot (verzi) dokumentu. Systém:

  1. Přijme soubor (DOCX / LaTeX)
  2. Převede ho na Markdown
  3. Asynchronně spustí pipeline kroky
  4. Porovná výsledné chyby s předchozí verzí — označí neopravené

Validace uploadovaného souboru (před konverzí):
  • Povolené typy: .docx a .tex — ověřit MIME type i příponu (ne jen příponu)
  • Maximální velikost souboru: 20 MB (FastAPI UploadFile limit v middleware)
  • Soubor se uloží na disk až po validaci — nikdy před ní
  • Pandoc se spouští s timeoutem (30 s) a pod omezeným uživatelem — ne jako root
Cascade delete — specifikace: DELETE /projects/{project_id} provede hard delete v tomto pořadí:
  1. Pokud existuje běžící pipeline job pro projekt → zastavit (ARQ job cancel)
  2. Smazat soubory z filesystemu / storage pro všechny verze projektu
  3. DB cascade: FOUND_ERROR → TOKEN_USAGE_EVENT → DOCUMENT_VERSION → PROJECT_PIPELINE_CONFIG → PROJECT
DELETE /versions/{version_id} (povoleno jen pro nejnovější verzi) — stejný postup pro danou verzi, bez smazání projektu.

Algoritmus porovnání chyb mezi verzemi

Prostý matching na char_start/char_end je křehký — při každé editaci textu se pozice posunou. Systém proto používá dvoustupňový matching:

1
Primary: context_snippet + error_type_slug
Každá chyba ukládá context_snippet — ~50 znaků textu okolí chyby (20 před, text chyby, 20 za). Pokud se snippet i slug shodují (Levenshtein vzdálenost ≤ 10 %), chyby jsou považovány za stejné.
2
Fallback: fuzzy pozice ± 200 znaků + slug
Pokud snippet nenajde shodu, zkusí se porovnání na základě overlapping char pozic s tolerancí ±200 znaků a shodným error_type_slug. Pokud ani to nestačí, chyba je označena jako nová.

Pipeline — stav analýzy

Pipeline běží asynchronně. Frontend používá polling — periodicky se ptá na stav, dokud se status nezmění na completed nebo failed.

GET /projects/{project_id}/versions/{version_id}/analysis Aktuální stav analýzy (polling) auth
POST /projects/{project_id}/versions/{version_id}/analysis/retry Opakovat analýzu (při selhání) auth
GET /faculties/{faculty_id}/pipeline-steps Dostupné kroky pipeline pro danou fakultu auth

GET /analysis — Response (průběh)

{
  "status": "processing",
  "progress": {
    "total_steps": 5,
    "completed_steps": 2,
    "current_step": "Kontrola citací",
    "percent": 40
  },
  "tokens_consumed_so_far": 4800,
  "started_at": "2026-03-31T14:22:05Z"
}
Pokud se v průběhu analýzy vyčerpá tokenový příděl uživatele, pipeline se zastaví a vrátí status: "partial" s reason: "token_limit_exceeded". Chyby nalezené do té doby jsou zachovány. Stav partial se liší od failed — analýza doběhla částečně, výsledky jsou použitelné.
Polling — specifikace chování frontendu:
  • Interval: 3 s prvních 30 s, pak 10 s (backoff po počátečním čekání)
  • Timeout: po 15 minutách bez terminálu stavu frontend přestane pollovat a zobrazí chybu
  • Na backendu: ARQ worker nastavuje status: failed pokud job neodpoví do 20 minut (ARQ job timeout) — zabraňuje uváznutí ve stavu processing
  • Terminální stavy: completed, partial, failed, conversion_failed — po jejich přijetí frontend polling zastaví

Chyby (Found Errors)

Chyba je základní jednotka výstupu analýzy. Má přesnou lokaci v textu (znaky), typ, závažnost a návrh řešení. UI na základě char_start a char_end zvýrazní příslušnou část Markdownu.

GET /errors/{error_id} Detail jedné konkrétní chyby auth
GET /error-types Seznam předdefinovaných typů chyb auth
IDOR ochrana na GET /errors/{error_id}: Endpoint musí projít celý řetězec vlastnictví: error → version → project → user_id == current_user.id. Nestačí zkontrolovat jen existenci záznamu.

FastAPI dependency pro chyby

async def get_error_for_user(
    error_id: UUID,
    current_user: User = Depends(get_current_user),
    db: Session = Depends(get_db)
) -> FoundError:
    error = db.get(FoundError, error_id)
    if not error:
        raise HTTPException(status_code=404)
    version = db.get(DocumentVersion, error.version_id)
    project = db.get(Project, version.project_id)
    if project.user_id != current_user.id:
        raise HTTPException(status_code=403)
    return error

Design pipeline

Pipeline je sada kroků, kde každý krok je jedno volání LLM s předpřipraveným promptem. Kroky jsou definovány per-fakulta superadminem a odrážejí specifické směrnice a pokyny jednotlivých fakult TUL pro psaní závěrečných prací.

Jak funguje jeden krok

0
Konverze souboru (Pandoc)
Po uploadu ARQ worker spustí Pandoc pro převod DOCX / LaTeX na Markdown. Maximální velikost dokumentu: 50 stran (~25–35k tokenů), která se vejde do context window. Při selhání konverze → status: conversion_failed, pipeline se nespustí.
1
Prompt template
Superadmin definuje prompt v admin dashboardu. Prompt tvoří system instrukci — obsahuje pouze instrukce pro LLM (co hledat, výstupní formát). Nikdy neobsahuje obsah dokumentu.
2
Sestavení vstupu — oddělení system / user
Markdown obsah dokumentu se posílá jako user message, nikdy jako součást system promptu. Tím se minimalizuje riziko prompt injection z obsahu práce studenta. Používá se JSON mode (structured output) pro deterministický výstup.
3
LLM volání + parsování
LLM vrátí JSON seznam chyb. Backend validuje výstup proti schématu. Nevalidní chyby jsou zahozeny. Timeout: 30 s per krok, retry: 3× s exponenciálním backoffem (1s, 2s, 4s). Při vyčerpání retry → krok označen jako failed, ostatní kroky pokračují.
4
Uložení a merge
Výstupy všech kroků se sloučí do jednoho seznamu chyb dané verze. Duplicity se deduplikují podle překrývajícího se rozsahu a stejného typu. Spotřeba tokenů se loguje per-krok do TOKEN_USAGE_EVENT.

Proč oddělené kroky místo jednoho volání?
Každý krok se specializuje na jeden typ problému, má vlastní prompt optimalizovaný pro daný kontext, a je snazší ho ladit, vylepšovat nebo vyměnit nezávisle. Zároveň umožňuje uživateli vybrat pouze relevantní kontroly. Kroky běží paralelně — každý jako samostatný ARQ sub-job. Selhání jednoho kroku neblokuje ostatní.
Prompt injection ochrana: System prompt (instrukce) a user content (dokument) jsou vždy předávány jako oddělené zprávy v LLM API volání. Používá se structured output / JSON mode — model vrací pouze validní JSON dle schématu, nikoli volný text. Obsah dokumentu studenta tak nemůže přepsat instrukce pipeline kroku.
Stavový přechod dokumentu: pending → processing → completed | partial | failed | conversion_failed
partial: pipeline doběhla, ale byl vyčerpán tokenový příděl před dokončením všech kroků — nalezené chyby jsou zachovány. conversion_failed: Pandoc nedokázal převést soubor, pipeline se nespustila.

Rate limiting

Spotřeba tokenů LLM je sledována per-uživatel. Systém má dva plány s různými příděly. Cílem je motivovat uživatele k upgradu na placený plán, aniž by byl free tier zcela nepoužitelný.

PlánSkenů / měsícPriorita fronty
institutional_free 5 (TBD po benchmarku) nižší
Basic 10 (TBD po benchmarku) střední
Pro 50 (TBD po benchmarku) vyšší
Konkrétní hodnoty přídělu jsou předmětem dalšího rozhodnutí. Doporučujeme je nastavit až po prvních real-world testech, kdy bude znám průměrný počet tokenů na analýzu. Orientačně: jedna analýza 50stránkového dokumentu (4 kroky) spotřebuje ~130k input + ~8k output tokenů.
Dvě oddělené metriky — neměsit je:
  • Sken = spuštění pipeline nad jednou DocumentVersion. Toto je primární uživatelský limit. Uložen jako monthly_scan_count INT v tabulce USER, resetován ARQ cron jobem 1. den každého měsíce. Před spuštěním pipeline: atomický UPDATE user SET monthly_scan_count = monthly_scan_count + 1 WHERE monthly_scan_count < scan_budget (jeden SQL příkaz, žádná race condition). Pokud UPDATE nezmění žádný řádek → limit dosažen, pipeline se nespustí.
  • TOKEN_USAGE_EVENT = cost monitoring pro admina. Každý krok loguje tokeny po dokončení. Uživatel limit tokenů nevidí — pouze procento skenů. Admin vidí reálné náklady per-uživatel.
  • Sloupce daily_token_budget a weekly_token_budget z tabulky USER se odstraňují — nahrazuje je monthly_scan_budget INT a monthly_scan_count INT.
Konkrétní hodnoty budgetu: nastavit empiricky po benchmarku (viz sekce Institucionální emaily). Orientačně: free = 5 skenů/měsíc, Basic = 10, Pro = 50.

Institucionální email — free tier bez zneužití

Free kredity jsou dostupné pouze pro ověřené institucionální emaily (např. @tul.cz). Každá univerzita má vlastní emailovou doménu, takže multi-account abuse je strukturálně nemožný — student má vždy jen jeden školní email. Běžné free emaily (gmail.com, seznam.cz apod.) free kredity nedostanou a musí přejít na placený plán.

Whitelist domén — datový model

Povolené domény jsou uloženy v tabulce ALLOWED_DOMAIN, kterou spravuje superadmin. Přidání nové univerzity = jeden řádek v tabulce, žádný deploy. 

ALLOWED_DOMAIN
  id                    UUID PK
  domain                VARCHAR(100) UNIQUE   -- "tul.cz", "cvut.cz", "muni.cz"
  university_name       VARCHAR(200)          -- "Technická univerzita v Liberci"
  free_analyses_per_month  INT DEFAULT 1      -- počet analýz zdarma
  enabled               BOOL DEFAULT TRUE
  created_at            TIMESTAMPTZ

Logika při registraci

1
Uživatel zadá email → systém extrahuje doménu (tul.cz).
2
Backend zkontroluje ALLOWED_DOMAIN. Nalezena shoda → User.tier = "institutional_free", free_analyses_per_month se nastaví dle záznamu. Nenalezena → User.tier = "unverified", bez free kreditů.
3
Na školní email se odešle verifikační odkaz (UUID token, platnost 24 h). Dokud uživatel email nepotvrdí, tier zůstává unverified a spuštění pipeline je blokováno.
4
Po kliknutí na odkaz → email_verified = TRUE, tier se aktivuje. Uživatel může spouštět analýzy.

Freemium logika — teaser místo truncation

Kritický designový princip: free tier neskrývá chyby náhodně, ale zobrazuje přesně 3 + ukáže počítadlo zbytku. Student vidí, že problém existuje, ale nezná lokaci ani detail — to je silnější motivace než „máš X skenů zdarma".

Příklad výstupu free skenování

✅ Sken dokončen — nalezeno 17 chyb

Zobrazuji 3 z 17:

  [CITACE] str. 12 — chybí rok vydání u zdroje "Novák et al."
  [FORMÁT] str. 4  — nadpis 2. úrovně není v souladu s normou ČSN
  [JAZYK]  str. 27 — pravděpodobná gramatická chyba ve větě "..."

  + 14 dalších chyb skryto — odemkni Pro pro úplný přehled
Proč nezobrazovat všechny chyby zdarma?
  • 5 skenů × „zobraz 3" = student vidí max 15 chyb. Průměrná BP má 20–60 chyb napříč kategoriemi — student stále neví, zda má vše opraveno.
  • Nestačí opravit 3 viditelné — skryté chyby dále existují. To vytváří přirozený tlak upgradovat.
  • Free model (Flash-Lite) navíc generuje stručnější, méně kontextuální popisy → kvalitativní rozdíl je viditelný i na těch 3 chybách.
